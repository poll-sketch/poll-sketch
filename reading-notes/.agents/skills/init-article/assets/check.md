# 文档质量检查规范 (Quality Checklist)

## 4.1 三级检查体系

### 一级：完整性检查 (Completeness)
- [ ] 标题是否包含核心关键词？（《边城》、软件工程、祖国发展）
- [ ] 摘要能否独立成篇？（不看正文是否完整）
- [ ] 每个提纲节点都有对应内容？（无悬空标题）
- [ ] 结论是否回应了摘要中提出的问题？
- [ ] 是否有明确的专业-文学关联分析？
- [ ] 是否涉及祖国发展需求的对接？

### 二级：一致性检查 (Consistency)
- [ ] 术语全文统一（如前文用"软件工程师"，后文不能用"程序员"指同一对象）
- [ ] 数据前后一致（如前文说"三个核心意象"，后文枚举是否为3个）
- [ ] 时态统一（回忆阅读用过去时，现状分析用现在时，规划用将来时）
- [ ] 人称统一（禁止"我/我们/笔者"混用，本规范建议用"我"）
- [ ] 标点规范（中文正文用全角标点，英文术语保留半角括号）
- [ ] 专业概念与文学意象的关联是否贯穿全文？

### 三级：有效性检查 (Effectiveness)
- [ ] 每章删除测试：删除该章是否损害整体理解？如否，则该章冗余
- [ ] 反向阅读测试：从结论倒推，论据是否足够支撑？
- [ ] 陌生人测试：假设读者是领域外人士，是否有概念未解释？
- [ ] 时间胶囊测试：三年后重读，是否仍能理解核心观点？
- [ ] 行动导向测试：读者读完后能采取什么具体行动？
- [ ] 专业相关性测试：是否避免了纯文学赏析，真正实现跨学科融合？

## 4.2 题材特化检查项

| 题材 | 特化检查项 |
|-----|----------|
| 软件工程报告 | 可复现性：关键配置/版本号/环境参数是否完整？安全性：是否泄露密钥/内网IP？ |
| 读书心得（本规范） | 原文引用占比：是否过度引用沦为摘抄？个人思考占比：感想是否依附于具体文本？专业融合度：是否真正将文学思想与专业学习结合？国家需求对接：是否明确阐述对祖国发展的意义？ |
| 入党申请书 | 政治表述准确性：党的性质/宗旨/指导思想引用是否精准？真实性：事例是否可查证？ |

## 4.3 评分与迭代机制
- **基线分**：满分100，一级检查每项12分，二级每项8分，三级每项6分
- **通过标准**：≥85分可输出；70-85分需标注待改进项；<70分重写
- **迭代指令**：若未通过，使用"基于[具体检查项]的反馈，请修正[具体位置]，要求[具体标准]"

## 4.4 常见失败模式 (Anti-patterns)
| 模式名称 | 症状 | 修复策略 |
|---------|------|---------|
| 头重脚轻 | 背景写3页，结论2句话 | 强制结论区字数≥摘要区×2 |
| 概念通货膨胀 | 堆砌术语无实质内容 | 每个术语必须跟"也就是说..."的解释句 |
| 情感透支 | 形容词密度>20% | 将所有"非常/极其/深刻"替换为具体程度副词或数据 |
| 正确的废话 | "我们要努力学习" | 加入主语、时间、衡量标准："我计划每天阅读30分钟专业文献" |
| 逻辑跳跃 | "因为A，所以C"缺B | 强制补充中间环节，或使用"这意味着..."过渡 |
| 两张皮现象 | 文学分析与专业内容互不关联 | 在每章添加"专业关联"小节，明确建立连接 |
| 口号堆砌 | "为祖国做贡献"而无具体行动 | 替换为可量化目标："参与开源项目贡献，提升国产软件影响力" |

## 4.5 专业融合质量标准
| 评价维度 | 优秀（5分） | 良好（4分） | 合格（3分） | 不合格（<3分） |
|---------|-----------|-----------|-----------|---------------|
| 关联深度 | 揭示本质联系 | 建立明确对应 | 表面类比 | 无关联 |
| 专业准确性 | 术语使用精准 | 术语使用正确 | 术语使用基本正确 | 术语使用错误 |
| 创新程度 | 提出新视角 | 提供新例证 | 常规关联 | 重复已有观点 |

## 4.6 字数合规检查
- [ ] 全文字数是否≥3000字？
- [ ] 各章节字数分配是否符合建议比例？
- [ ] 专业论述部分是否≥全文40%？
