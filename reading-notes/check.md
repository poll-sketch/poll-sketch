# 文档质量检查规范 (Quality Checklist)

## 4.1 三级检查体系

### 一级：完整性检查 (Completeness)

- [ ] 标题是否包含"边城"关键词？
- [ ] 摘要能否独立成篇？（300字内概括阅读背景、核心感受、结构安排）
- [ ] 每个提纲节点都有对应内容？（无悬空标题）
- [ ] 结论是否回应了摘要中提出的问题？
- [ ] 原文引用是否标注了出处（章回/页码）？

### 二级：一致性检查 (Consistency)

- [ ] 术语全文统一（如"茶峒"不可一会写作"茶洞"）
- [ ] 人物关系一致（翠翠爷爷的身份、傩送与天保的兄弟关系）
- [ ] 时态统一（回忆阅读感受用过去时，当前反思用现在时）
- [ ] 人称统一（以"我"为主，禁止"我/我们/笔者"混用）
- [ ] 字数符合要求（约3000字，浮动±10%）

### 三级：有效性检查 (Effectiveness)

- [ ] 每章删除测试：删除"个人经验勾连"章节是否损害整体理解？如否，则该章冗余
- [ ] 反向阅读测试：从结论倒推，论据是否足够支撑"自然人性与命运悲剧交织"的命题？
- [ ] 陌生人测试：假设读者从未读过《边城》，是否能从本文获得完整理解？
- [ ] 时间胶囊测试：三年后重读本文，是否仍能理解核心观点？
- [ ] 行动导向测试：读者读完后能否明确知道这本书如何影响了我？

## 4.2 题材特化检查项

| 检查维度 | 读书心得特化检查项 |
|---------|------------------|
| 原文引用占比 | 是否过度引用沦为摘抄？（引用应<15%，其余为个人思考） |
| 个人思考占比 | 感想是否依附于具体文本？（禁止空谈"人性的美好"而无文本支撑） |
| 身份特征融合 | 是否体现了"软件工程学生"的独特视角？（如联系代码/算法/团队协作等） |
| 情感真实性 | 是否避免了泛化的"催人泪下"式抒情？（需具体指出触发的场景） |

## 4.3 评分与迭代机制

- **基线分**：满分100，一级检查每项15分，二级每项10分，三级每项5分
- **通过标准**：≥85分可输出；70-85分需标注待改进项；<70分重写
- **迭代指令**：若未通过，使用"基于[具体检查项]的反馈，请修正[具体位置]，要求[具体标准]"

## 4.4 常见失败模式 (Anti-patterns)

| 模式名称 | 症状 | 修复策略 |
|---------|------|---------|
| 头重脚轻 | 书籍介绍写2页，感想只写3段 | 强制感想部分≥70%篇幅 |
| 概述替代感想 | 大量复述剧情，缺乏个人分析 | 每概述一段剧情后，必须跟一段个人反思 |
| 身份脱节 | 读书心得与软件工程身份毫无关联 | 必须在"个人经验勾连"部分明确联系专业背景 |
| 泛化抒情 | "读完很感动"、"很有意义"等空洞表达 | 替换为具体场景：如"读到翠翠等在渡口，我想起自己等待面试结果的那个下午" |
| 逻辑跳跃 | "因为《边城》很美，所以我们要保护传统" | 强制补充中间推理环节 |

## 4.5 最终输出格式

通过检查后，文章应保存为：

```
d:\JAVA\IDE\venv\reading-notes\边城读书心得.md
```

格式要求：
- 使用 Markdown 语法
- 标题层级清晰（一级标题为文章标题，二级标题为章节）
- 段落之间空一行
- 重点语句可用**粗体**标注，但不宜过多
